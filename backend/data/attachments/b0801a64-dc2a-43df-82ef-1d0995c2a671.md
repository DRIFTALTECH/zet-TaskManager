# 02 — Skills Agent

> The productised, agent‑scoped version of [01 — the Skill Engine](01-agentstudio-skill-engine.md).
> This is what the Driffy gateway routes most chat traffic to. It adds: per‑agent
> skill rosters, always‑on global tools, extended thinking, per‑tool retries, and
> **live WebSocket streaming**.

## Purpose

A "skills agent" is a configurable agent stored in `agents.skills_agents`. It owns
a curated subset of skills (not the whole `skill_instances` table) and a set of
global tools that are bound on every turn. A **supervisor** (orchestrator) decides
whether to answer directly, call a global tool, or hand off to the **executor** to
run a selected skill. Tool failures retry up to 3× before escalating back to the
supervisor.

## Config shape (`load_skills_agent`)

Reads `agents.skills_agents` by slug `name`, then two link tables:

```
skills_agents (id, name, display_name, system_prompt,
               thinking_enabled, thinking_budget_tokens, max_turns,
               tracing_enabled, orchestrator_agent_id, executor_agent_id, status)
   ├─ skills_agent_skill_links → skill_instances (+ base skills) → cfg["skills"]
   └─ skills_agent_tool_links  → tool_instance_id[]              → cfg["global_tool_instance_ids"]
```

Raises `ValueError` if missing or `status != 'active'`. The returned dict is the
single source of truth passed through the whole run via a contextvar.

## Data flow

```
run_skills_agent(query, cfg, ws_emitter, history, session_id)
        │  set contextvars: _AGENT_CONFIG, _WS_EMITTER, TRACING_ENABLED, _EXECUTOR_STEP
        ▼
START ─► orchestrate ──┬─(no tool_calls)──────────────► END
   ▲                   ├─(select_skill only)─► prepare_execution ─► execute
   │                   └─(global tool call)──► supervisor_tools ─┐
   │                                                             │
   └──────────── report ◄─ retry_check ◄─ tools ◄─ execute ◄─────┘
                    │            │  (≤3 retries → execute; escalate → orchestrate)
                    └──► orchestrate (loop)
```

### Nodes (`runtime.py`)

- **`orchestrate_node`** (the supervisor). LLM = `cfg.orchestrator_agent_id`
  (default 9). Tools bound = **global tools** + a synthetic `skills_agent_select_skill`
  tool. The system message (built once, cached) contains the UI system prompt +
  `## Available Skills` (id/name/description only) + `## Available Tools`
  (name/desc/params). It `bind_tools(...)` and invokes. `is_complete` = the model
  produced **no** tool calls.
- **`_route_orchestrate`** decides the next node:
  - no tool calls → `done` (END)
  - only `select_skill` → `prepare_execution`
  - any other tool call → `supervisor_tools`
- **`supervisor_tools_node`** runs the global tools (not skill tools, not
  `select_skill`), emits `tool_call_start`/`tool_call_result` WS events, returns
  `ToolMessage`s, loops back to `orchestrate`.
- **`prepare_execution_node`** extracts the `select_skill` call, loads the chosen
  skill from `cfg["skills"]`, **resets the message tape** (`RemoveMessage` + prior
  history + user query + previous step result), bumps `run_count`, resets retry
  counters, emits `skill_selected`.
- **`execute_node`** (the executor). LLM = `cfg.executor_agent_id` (default 16).
  Tools bound = **skill tools only** (`build_tools_for_skill_instance`). System
  prompt = the executor agent's `agents.agents.prompt` + the skill's instructions.
  - If `cfg.thinking_enabled` and the backend supports it
    (`detect_thinking_backend`), binds thinking kwargs.
  - **Streaming**: if a `_WS_EMITTER` is set, calls
    `sync_stream_llm_step(bound_llm, msgs, on_stream_chunk, on_event)` which streams
    tokens live on `thinking`/`response` channels and sets `_RESPONSE_STREAMED_LIVE`.
    Otherwise a plain `.invoke()`.
- **`tools_node`** runs skill tools **and** global tools (both available to the
  executor). Special: for Argus skills it enriches args
  (`enrich_argus_tool_arguments`) and records payloads
  (`record_tool_payload`) for the turn‑finalize step. Emits per‑tool WS events.
- **`retry_check_node`** scans the latest `ToolMessage`s for `success:false`.
  - failures and `tool_retry_count < 3` → bump counter, emit `retry`, route back
    to `execute`.
  - failures at the cap → build a failure context, emit `escalate`, route to
    `orchestrate` so the supervisor can pick a different skill.
  - no failures → `report`.
- **`report_node`** appends `{skill_name, output}` to `skill_results`, loops to
  `orchestrate`.

### `_run_graph` finalisation

After the graph completes, `_run_graph` extracts the last plain AIMessage as the
reply. For **Argus** skill instances it may *force* a finalised reply via
`finalize_argus_skill_reply(...)` (turn‑finalize handler). Returns
`{response, skill_name, run_count, is_complete, response_streamed_live}`.

## WebSocket protocol (`ws_router.run_websocket`)

```
client/gateway ──► WS /api/skills-agents/{name}/ws
   parse_inbound_frame(raw) → ParsedTurn{text, history, control, session_id}
      control ping → {type:pong}
      control terminate/disconnect → ack + break
      control connect/invalid → continue
   else: run_skills_agent in a thread executor; events flow through an asyncio.Queue
         each event → normalize_graph_event(event) → websocket.send_json(frame)
   on completion:
      if not streamed live → send the reply as stream_chunk(channel=response) + done
      send message_response (skill-instance / gateway envelope)
      send legacy {type:done, response, skill_name, run_count}
```

### Frame types emitted

| Frame | When | Source |
|-------|------|--------|
| `stream_chunk` (channel=`thinking`/`response`) | live token deltas | `sync_stream_llm_step` |
| `llm_start` / `llm_end` / `thinking_done` / `response_promote` | step boundaries | `core/ws_llm_streaming` |
| `tool_call_start` / `tool_call_result` | per tool | `tools_node` / `supervisor_tools_node` |
| `skill_selected`, `retry`, `escalate` | lifecycle | runtime nodes |
| `message_response` | final reply (gateway persists this) | `ws_protocol.send_message_response` |
| `done` | legacy direct clients | `ws_router` |

`parse_inbound_frame` handles **both** direct clients (`{"message":...,"history":[]}`
or plain text) and the **Driffy gateway** upstream shape
(`{"history":[...],"msg":"current_user_context: ...\nmsg: ..."}`). It unwraps the
gateway `msg` wrapper, strips heavy skill‑card envelopes from history into
breadcrumbs (token‑budget hygiene), and extracts `session_id`.

## File‑by‑file connections

```
ws_router.py
  ├─ loader.load_skills_agent(name)            → cfg
  ├─ runtime.run_skills_agent(query, cfg, ...) → result   (in a thread)
  ├─ ws_protocol.parse_inbound_frame / send_*  → frame I/O
  ├─ core.ws_llm_streaming.normalize_graph_event → event → frames
  └─ api.skills_agent_crud.*                   → CRUD endpoints

runtime.py
  ├─ agentstudio.agent.skill_loader.build_tools_for_skill_instance / _instance_ids
  ├─ agentstudio.core.llm_init.get_llm_for_agent_id
  ├─ agentstudio.core.tool_card_payload.llm_facing_tool_output
  ├─ core.llm_cache.cached_system_message
  ├─ core.ws_llm_streaming.sync_stream_llm_step
  ├─ .thinking.*                               → extended thinking
  ├─ .trace_context.* / .trace_payload.*       → tracing per node
  └─ tracing.tracer.tracer                     → @_tracer decorators
```

Note the engine reuse: the Skills Agent **does not** re‑implement tool building —
it imports `build_tools_for_skill_instance` and `build_tools_for_instance_ids`
straight from the Skill Engine's `skill_loader.py`.

## DB tables

| Table | For |
|-------|-----|
| `agents.skills_agents` | the agent config (prompt, thinking, agent ids). |
| `agents.skills_agent_skill_links` → `agents.skill_instances` (+ `agents.skills`) | the agent's skill roster. |
| `agents.skills_agent_tool_links` → `agents.tool_instances` | always‑on global tools. |
| (then everything from the Skill Engine for executing a skill) | tools. |

## Entry points

| Route | Handler |
|-------|---------|
| `GET/POST/PATCH/DELETE /api/skills-agents[/{name}]` | CRUD (`skills_agent_crud`). |
| `POST /api/skills-agents/{name}/run` | blocking run, returns `{response, skill_name, run_count, is_complete, events}`. |
| `WS /api/skills-agents/{name}/ws` | streaming run (the gateway's main target). |
